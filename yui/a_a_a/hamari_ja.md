サンプルコード作成中に遭遇したパーサ／ランタイムの落とし穴と回避策のメモ。

## 1. (修正済み) `もし <式1> が <式2> ...ならば` の左辺に関数呼び出しを直接書けない

commit `67bcc99` (fix(parser): パーサー既知バグ 3 件の修正) で修正済み。
以下はそれ以前の回避策メモ。

### 症状 (過去)

```
もし剰余(j,2)が0ならば{
   ...
}
```

`expected-expression` でパースエラーになっていた。
`extract_special_names` が `もし剰余(` を 1 名として登録してしまい、
パーサが `剰余(j,2)` の直後の `が` を式の継続として読めなかったため。

### 原因と修正

`syntax.json` に新キー `special-name-exclude-prefix` を追加し、
`もし`/`ならば`/…/`[0-9]+` の接頭辞を剥離した版も special-name として登録するようにした。

### (参考) かつての回避策

いったん変数に束ねてから比較すれば動いた。

```
parity=剰余(j,2)
もしparityが0ならば{
   ...
}
```

現在は `もし剰余(j,2)が0ならば{...}` のように直接書いても通る。

## 2. 算術記号 `+ - * / %` は yui 構文では使えない

`a+b` のような書き方は無効。標準ライブラリの関数を使う:

| やりたいこと | 書き方 |
|--------------|--------|
| `a + b`      | `和(a,b)` |
| `a - b`      | `差(a,b)` |
| `a * b`      | `積(a,b)` |
| `a / b` (整数商) | `商(a,b)` |
| `a % b`      | `剰余(a,b)` |

ループのインデックス計算 (`i+1`, `n-1`) も同様で、`和(i,1)` / `差(n,1)` と書く。
冒頭に `標準ライブラリを使う` を忘れないこと。

---

## 3. 比較は記号ではなく接尾辞

`==` `!=` `<` `>` `<=` `>=` は使えない。すべて `もし...が...<接尾辞>ならば` の形に置き換える。

| 比較 | 接尾辞 |
|------|--------|
| `==` | (なし) |
| `!=` | `以外` |
| `<`  | `より小さい` |
| `>`  | `より大きい` |
| `<=` | `以下` |
| `>=` | `以上` |
| `in` | `のいずれか` |
| `not in` | `のいずれでもない` |

## 4. 配列から要素を「削除」する手段がない (pop / shift がない)

### 症状

スタックや動的配列を素朴に実装すると、`追加する` で末尾を伸ばすことはできるが、
**末尾を取り除く** プリミティブがない。論理サイズだけ管理する変数 (`top`) を
別に持って「論理 pop」したつもりでも、その後の `追加する` は配列の実末尾に
連結されてしまい、論理 top と実長がずれて値が壊れる。

### 例 (壊れる)

```
stack=[]; top=0
stackに3を追加する; topを増やす     # stack=[3], top=1
stackに4を追加する; topを増やす     # stack=[3,4], top=2
topを減らす; topを減らす             # 論理 pop, top=0  (stack はまだ [3,4])
stackに7を追加する; topを増やす     # stack=[3,4,7], top=1
                                       # ↑ stack[top-1]=4 になってしまう (期待は 7)
```

### 回避策

push 側で「論理 top が配列実長より小さければ上書き、そうでなければ追加」と分岐する。

```
sz=stackの大きさ
もしtopがszより小さいならば{
   stack[top]=x
}
そうでなければ{
   stackにxを追加する
}
topを増やす
```

### ルール

論理サイズと配列実長が独立に動くデータ構造 (スタック・リングバッファ・
可変長キュー) では、**push を必ず「上書き or 追加」の分岐で書く**。
あるいは「論理 pop が起きない設計」(常に末尾追加・末尾参照のみ) に絞る。

## 5. `商(int, int)` は床除算 → 小数の答えがほしいときは明示的に `小数化`

### 症状

二次方程式の解の公式や確率計算など、整数の入力から **小数の答え** を出したい
ときに `商(差(0,b), 積(2,a))` と素直に書くと、整数同士の床除算になって
結果が壊れる (例: `商(3, 2)` は `1.500000` ではなく `1`)。

### 回避策

割られる側・割る側のどちらかを `小数化` で float に上げてから `商` に渡す。
両方上げておけば確実。

```
denom=小数化(積(2,a))
mb=小数化(差(0,b))
x1=商(和(mb,sqrtD),denom)
```

### ルール

- 整数 OP 整数 → 整数 (`商` は床除算)
- 整数 OP 小数 → 小数 (一方が float なら昇格)
- 小数の答えがほしい計算では、入力段階で `小数化` を挟むのが安全

## 6. (修正済み) doctest 期待値直後のコメント中の `<1桁数字> (...)` が関数呼び出しに化ける

commit `67bcc99` で修正済み。`extract_special_names` の前にコメントを除去するようにした。
以下はそれ以前の観測メモ。

### 症状 (過去)

`>>> 式 / 期待値` の **直後** のコメントに `<1桁の数字> <空白> (...)` の
パターンがあると、その `数字(...)` が関数呼び出しとして解釈され、
`undefined-variable: <数字>` で実行時エラーになる。

エラー位置は壊れたコメント行ではなく、その後の `>>> 式` 内のリテラルとして
報告される (例: `lshift(1,3)` の `3` が「未定義変数」扱い)。原因の特定が
極めて困難。

### 観測した最小再現

```
f=入力xに対し{
   xが答え
}
>>> f(3)
3
# 3 (foo)            ← この `3 (` が関数呼び出しに化ける
>>> f(3)
3
```

### トリガー条件 (実験で判明)

| コメント                              | 結果 |
|---------------------------------------|------|
| `# X (端数切り捨て)`                  | OK   |
| `# X 3 (端数切り捨て)`                | FAIL |
| `# 3 (foo)`                           | FAIL |
| `# 3 (端数切り捨て)`                  | FAIL |
| `# 13 (端数切り捨て)`                 | OK   ← 多桁数字なら通る |
| `# 13 を 2 ビット右シフトすると 3 (端数切り捨て)` | FAIL ← 末尾の `3 (` が刺さる |
| `# 13 を 2 X (端数切り捨て)`          | OK   ← 数字の直後を文字で区切れば通る |
| `# 13 を 2 (端数切り捨て)`            | OK   ← 多桁 `13` で `2 (` の前に何かあれば通る |

直前に **少なくとも 1 つの `>>> 式 / 期待値` ブロックがある** ことが必要。
パーサ単体ではなく、doctest assertion との相互作用で壊れる。

(以前は「`N=10` や `13 を 2` 系のキーワードが原因」と推測していたが、
`yuichan/test_parser_bugs.py` で再調査した結果、本当のトリガーは
**1 桁数字 + 空白 + 開き括弧** であると判明した)

### 推測される原因

doctest assertion の期待値解析が改行で止まらず、直後のコメント行を
`#` 剥がしせずに次の式の一部として読んでしまっている疑い。
結果として `3\n# 3 (foo)\n` が `3 (foo)` のような関数呼び出しに化ける。
詳細は未追跡。

### 回避策

doctest の期待値直後のコメントに **`<数字> (...)`** を書かない:

```
# 端数は切り捨てられる            ← OK (数字なし)
# 13 (端数切り捨て)               ← OK (多桁数字)
# 切り捨てて 3 になる             ← OK (数字の直後に括弧がない)
# 3 (端数切り捨て)                ← NG (1桁数字 + 空白 + 括弧)
```

迷ったら **コメントから数字と括弧の組み合わせを排除** する。
コードや式の例はファイル冒頭の (doctest を含まない) ブロックにまとめる。

## 7. `continue` が無い → 条件反転 or スキップフラグで書き換える

### 症状

ループ本体で「ある条件のとき残りをスキップして次の反復へ」と書きたいが、
yui には `continue` 相当のキーワードが無い (`くり返しを抜ける` は `break`)。

### 回避策 A: 条件反転 + ネスト もし

スキップ条件を反転して、本体を入れ子の `もし` の中に入れる。

```
n回くり返す{
   x=arr[i]
   もしxが0以上ならば{        # x < 0 のときの continue を反転
      もしxが100以下ならば{    # x > 100 のときの continue を反転
         # 本体
      }
   }
   iを増やす
}
```

利点: フラグ不要で読みやすい。
欠点: スキップ条件が増えるとネストが深くなる。

### 回避策 B: スキップフラグ

ループの先頭で `skip=0` を置き、continue したい所で `skip=1` を立て、
本体を `もしskipが0ならば` で囲む。

```
n回くり返す{
   x=arr[i]
   skip=0
   もしxが0より小さいならば{ skip=1 }
   もしxが100より大きいならば{ skip=1 }
   もしskipが0ならば{
      # 本体
   }
   iを増やす
}
```

利点: スキップ条件が増えてもネストが平ら。
欠点: フラグ変数が必要。

### ルール

スキップ条件が 1〜2 個なら **A (条件反転)**、3 個以上または条件式が複雑なら
**B (スキップフラグ)** を使うと読みやすい。

## 8. (取り下げ) `もし <変数> が <変数> ならば` は実は壊れていない

最初は「両辺が変数で suffix なしの `もし` がパース失敗する」と記録していたが、
`yuichan/test_parser_bugs.py` で再調査した結果、これは独立したバグではなく、
**Bug 9 (1 行形式 + 増減文)** が body に `eqを増やす` を持つ場面で先に
表面化していただけと判明した。

```
もしaがbならば{ x=1 }       ← OK     (代入文ボディ)
もしaがbならば{ eqを増やす } ← NG     (Bug 9)
```

両辺変数 + suffix なし自体は問題ない。詳細は Bug 9 を参照。

## 10. `もし <式> が <関数呼び出し> [接尾辞] ならば` — 右辺の関数呼び出しがパース失敗

### 症状

`もし` 比較の **右辺**に関数呼び出しを直接書くとパースエラーになる。
接尾辞の有無に関係なく発生する。

```
もしposが差(0,1)ならば{ ... }         ← FAIL: expected-token ❌`ならば` ✅`が` 🧬if-infix
もし1が差(0,1)より大きいならば{ ... } ← FAIL: wrong-name ❌差(0,1)より大きいならば
```

**左辺**の関数呼び出しはハマり #1 (`67bcc99` で修正済) で通るようになったが、
右辺の関数呼び出しは同じ修正が効いておらず、別経路で引っかかっている模様。

### 最小再現

```
標準ライブラリを使う
pos=差(0,1)
もしposが差(0,1)ならば{ "hit" }   ← FAIL
```

### 回避策

右辺をいったん変数に束ねてから比較する。

```
neg1=差(0,1)
もしposがneg1ならば{
   ...
}
```

### ルール

`もし A が B ならば` (および `...より大きい/小さい/以上/以下/以外`) の
**B が関数呼び出しになる場合は、先に変数へ束ねる**。リテラル・変数・インデックス式
であれば問題なく通る。

## 9. (修正済み) 1 行形式 `もし...ならば{ xを増やす }` がパース失敗する (xを減らす は通る)

commit `67bcc99` で修正済み。`capture_line` の停止トークンに `block-begin` を追加し、
lookahead がインラインブロック内部の「増やす」を見ないようにした。
以下はそれ以前の観測メモ。

### 症状 (過去)

1 行形式 `もし...ならば{ ... }` の本体に `xを増やす` を書くと
`expected-expression` でパース失敗する。
**対称的なはずの `xを減らす` は同じ位置で動く** ── 増やす だけが壊れている。

```
もし1が1ならば{ xを増やす }    ← FAIL (expected-expression)
もし1が1ならば{ xを減らす }    ← OK   (なぜか通る!)
もし1が1ならば{ x=和(x,1) }    ← OK   (代入文は通る)
```

### 観測

- 左辺が変数か リテラルかは無関係 (`もしaがbならば{ xを増やす }` も同じく FAIL)
- `xを増やす` の代わりに `xを減らす` にすれば動く (非対称!)
- 代入文ボディ (`{ x=1 }`, `{ x=和(x,1) }`) は問題なく通る

### 回避策

複数行に展開する。多行形式なら `xを増やす` も問題なく動く。

```
もしdabが0ならば{
   eqを増やす
}
```

### ルール

迷ったら **本体は必ず複数行に書く**。1 行形式は短い代入文の場合のみ使う。
特に `xを増やす` を 1 行形式に書きたくなったら、必ず多行に展開する。
